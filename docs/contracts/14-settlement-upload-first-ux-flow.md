# 14. Settlement Upload-First UX Flow

## 문서 목적

이 문서는 `front-web-console`의 `정산` 그룹을 upload-first 운영 흐름 기준으로 고정한다.

이번 문서는 아래를 먼저 결정한다.

1. 배차 업로드와 정산 입력의 경계
2. 회사/플릿 문맥을 페이지 사이에서 어떻게 유지하는지
3. `입력 검토 -> 실행 -> 결과`가 어떻게 이어지는지
4. 개별 생성이 어디까지 예외인지

## 기본 원칙

1. 배차표 업로드의 시작점은 `배차 보드 상세`다.
2. 정산 입력은 기본적으로 사람이 한 줄씩 직접 만드는 화면이 아니다.
3. `정산 처리` 안에서는 현재 작업 중인 `company / fleet` 문맥이 계속 유지돼야 한다.
4. `정산 실행`은 배차 확정과 snapshot bootstrap이 끝난 입력이 있을 때만 다음 단계로 넘어간다.
5. `정산 결과`는 실행 이후의 읽기/검토 화면이다.

## 1. 그룹 공통 문맥

### 1.1 유지해야 하는 값

1. `company`
2. `fleet`

### 1.2 유지 방식

1. 사용자가 `정산 처리` 안에서 페이지를 옮겨도 위 문맥은 유지된다.
2. `정산 기준`, `정산 입력`, `정산 실행`, `정산 결과`는 같은 문맥을 공유해야 한다.
3. 문맥이 비어 있으면 `정산 조회`를 제외한 write 화면은 먼저 문맥 선택부터 요구한다.

## 2. 페이지 역할

### 2.1 `정산 조회`

1. 현재 문맥의 최근 실행 상태, 최근 결과, 예외 여부를 읽는다.
2. write 시작점이 아니라 현재 상태 확인용 landing page다.

### 2.2 `정산 기준`

1. 전역 정산 설정과 현재 `company / fleet`의 단가표를 확인한다.
2. 전역 세금/공제/보험과 회사·플릿 운영 단가를 같은 패널에 섞지 않는다.
3. 이 페이지에서 정산 입력을 직접 만들지 않는다.

### 2.3 `정산 입력`

1. 배차 업로드에서 확정된 batch와 그로부터 생성된 `delivery record + daily snapshot`을 먼저 보여준다.
2. 기본 시나리오는 `배차 업로드 결과 검토 -> snapshot 검토 -> 예외 보정 -> 실행으로 진행`이다.
3. 수기 생성은 예외 보정용 보조 액션으로만 남긴다.
4. 현재 문맥의 batch 수, record 수, snapshot 수, 총 박스 수를 먼저 읽히게 둔다.

### 2.4 `정산 실행`

1. 이 페이지는 유효한 snapshot과 현재 회사·플릿 단가표를 기준으로 run을 만든다.
2. 기본 CTA는 `정산 실행 생성`이다.
3. 유효한 snapshot이 없거나 단가표가 없으면 실행 CTA는 비활성화한다.
4. 실행 후에는 바로 `정산 결과`로 이어지는 진입을 제공한다.

### 2.5 `정산 결과`

1. 생성된 run과 item을 검토한다.
2. 결과 페이지는 검토와 예외 확인이 기본 목적이다.
3. 결과 원본을 사람이 한 줄씩 만드는 화면으로 보지 않는다.

## 3. 개별 생성 규칙

1. 개별 생성은 기본 경로가 아니다.
2. 개별 생성은 `예외 보정`일 때만 연다.
3. 개별 생성이 필요해도 full-page 생성 화면으로 열지 않는다.
4. 개별 생성은 그룹 페이지 안의 보조 모달로만 연다.
5. `정산 입력`에서 개별 레코드 생성은 기본 CTA보다 뒤에 둔다.
6. `정산 결과`에서 개별 item 생성은 1차 운영 UX의 기본 경로로 두지 않는다.

## 4. 단계 연결 규칙

1. `정산 조회`는 독립 read 진입이다.
2. 실제 운영 흐름의 시작점은 보통 `배차 보드 상세` 또는 `정산 기준 / 정산 입력`이다.
3. `정산 입력`이 끝나면 `정산 실행`으로 자연스럽게 이동해야 한다.
4. `정산 실행`이 끝나면 `정산 결과`로 자연스럽게 이동해야 한다.
5. `정산 처리` 화면 안에서는 탭 클릭과 다음 단계 CTA를 함께 제공한다.

## 5. UI 구성 규칙

### 5.1 `정산 입력`

1. 첫 화면에서 현재 문맥의 upload-first review 요약이 가장 먼저 보여야 한다.
2. 확정 upload batch, 생성된 record, active snapshot이 같은 문맥 안에서 이어서 보여야 한다.
3. 기존 개별 생성 폼이나 개별 생성 모달은 보조 액션으로 내려간다.

### 5.2 `정산 실행`

1. 현재 문맥
2. 전역 정산 설정 요약
3. 회사·플릿 단가표 요약
4. 최근 snapshot 요약
5. 실행 CTA

위 순서로 읽히게 둔다.

### 5.3 `정산 결과`

1. 현재 run 요약
2. 결과 item 목록
3. 오류/예외 요약

위 순서로 읽히게 둔다.

## 6. Shared Read 경계

1. `operator`는 이 문서의 write 흐름을 갖지 않는다.
2. `operator`는 계속 `/api/settlement-ops/` 기반 read-only summary만 본다.
3. `admin`도 `정산 조회` 페이지에서는 read-only로 본다.

## 7. 배차 handoff 규칙

1. `배차 보드 상세`가 배차표 업로드 preview/confirm을 소유한다.
2. 확정 upload batch는 `service-delivery-record` snapshot bootstrap의 원천이 된다.
3. `정산 입력`은 배차가 만든 결과를 검토하고 예외만 보정한다.

## 8. 현재 코드와의 관계

1. 현재 `front-web-console`은 좌측 `정산 조회 / 정산 처리` 분리와 process 탭 구조를 이미 반영한다.
2. `배차 보드 상세`는 upload preview/confirm과 snapshot bootstrap 진입을 소유한다.
3. `정산 입력`은 upload-first review screen으로 동작한다.
