# 14. Settlement Upload-First UX Flow

## 문서 목적

이 문서는 `front-admin-console`의 `정산` 그룹을 운영형 흐름 기준으로 다시 고정한다.

이번 문서는 아래를 먼저 결정한다.

1. `정산 입력`의 기본 경로가 무엇인지
2. 회사/플릿 문맥을 페이지 사이에서 어떻게 유지하는지
3. `입력 -> 검증 -> 실행 -> 결과`가 어떻게 이어지는지
4. 개별 생성이 어디까지 예외인지

## 기본 원칙

1. `정산 입력`의 기본 경로는 `엑셀 업로드`다.
2. 정산 입력은 기본적으로 사람이 한 줄씩 직접 만드는 화면이 아니다.
3. `정산` 그룹 안에서는 현재 작업 중인 `company / fleet` 문맥이 계속 유지돼야 한다.
4. `정산 실행`은 검증된 입력이 있을 때만 다음 단계로 넘어간다.
5. `정산 결과`는 실행 이후의 읽기/검토 화면이다.

## 1. 그룹 공통 문맥

### 1.1 유지해야 하는 값

1. `company`
2. `fleet`
3. 필요하면 `period`

### 1.2 유지 방식

1. 사용자가 `정산` 그룹 안에서 페이지를 옮겨도 위 문맥은 유지된다.
2. `정산 기준`, `정산 입력`, `정산 실행`, `정산 결과`는 같은 문맥을 공유해야 한다.
3. 문맥이 비어 있으면 `정산 조회`를 제외한 write 화면은 먼저 문맥 선택부터 요구한다.

## 2. 페이지 역할

### 2.1 `정산 조회`

1. 현재 문맥의 최근 실행 상태, 최근 결과, 예외 여부를 읽는다.
2. write 시작점이 아니라 현재 상태 확인용 landing page다.

### 2.2 `정산 기준`

1. 현재 `company / fleet`에 연결된 정책과 버전을 확인한다.
2. 정책 연결을 수정할 수 있다.
3. 이 페이지에서 정산 입력을 직접 만들지 않는다.

### 2.3 `정산 입력`

1. 기본 CTA는 `엑셀 업로드`다.
2. 업로드 후에는 바로 저장 완료로 끝내지 않고 `검증 요약`을 먼저 보여준다.
3. 검증 요약에는 아래가 보여야 한다.
   - 총 행 수
   - 유효 행 수
   - 오류 행 수
   - 중복 여부
   - 누락된 필수값 수
4. 사용자는 검증 요약을 보고 `실행으로 진행` 또는 `업로드 다시 하기`를 선택한다.

### 2.4 `정산 실행`

1. 이 페이지는 검증 통과한 입력 배치로만 run을 만든다.
2. 기본 CTA는 `정산 실행 생성`이다.
3. 유효한 입력 배치가 없으면 실행 CTA는 비활성화한다.
4. 실행 후에는 바로 `정산 결과`로 이어지는 진입을 제공한다.

### 2.5 `정산 결과`

1. 생성된 run과 item을 검토한다.
2. 결과 페이지는 검토와 예외 확인이 기본 목적이다.
3. 결과 원본을 사람이 한 줄씩 만드는 화면으로 보지 않는다.

## 3. 개별 생성 규칙

1. 개별 생성은 기본 경로가 아니다.
2. 개별 생성은 `예외 보정`일 때만 연다.
3. 개별 생성이 필요해도 full-page 생성 화면으로 열지 않는다.
4. 개별 생성은 그룹 페이지 안의 보조 모달로만 연다.
5. `정산 입력`에서 개별 레코드 생성은 기본 CTA보다 뒤에 둔다.
6. `정산 결과`에서 개별 item 생성은 1차 운영 UX의 기본 경로로 두지 않는다.

## 4. 단계 연결 규칙

1. `정산 조회 -> 정산 기준 -> 정산 입력 -> 정산 실행 -> 정산 결과` 순서는 유지한다.
2. 하지만 실제 운영 흐름의 시작점은 보통 `정산 입력`이다.
3. `정산 입력`이 끝나면 `정산 실행`으로 자연스럽게 이동해야 한다.
4. `정산 실행`이 끝나면 `정산 결과`로 자연스럽게 이동해야 한다.
5. 단계 카드 클릭은 남기되, 각 페이지 안에서 다음 단계 진입 CTA도 제공한다.

## 5. UI 구성 규칙

### 5.1 `정산 입력`

1. 첫 화면에서 `엑셀 업로드` 영역이 가장 먼저 보여야 한다.
2. 업로드 결과와 검증 요약은 같은 문맥 안에서 이어서 보여야 한다.
3. 기존 개별 생성 폼이나 개별 생성 모달은 보조 액션으로 내려간다.

### 5.2 `정산 실행`

1. 현재 문맥
2. 연결된 정책 요약
3. 최근 업로드/검증 요약
4. 실행 CTA

위 순서로 읽히게 둔다.

### 5.3 `정산 결과`

1. 현재 run 요약
2. 결과 item 목록
3. 오류/예외 요약

위 순서로 읽히게 둔다.

## 6. Shared Read 경계

1. `operator`는 이 문서의 write 흐름을 갖지 않는다.
2. `operator`는 계속 `/api/settlement-ops/` 기반 read-only summary만 본다.
3. `admin`도 `정산 조회` 페이지에서는 read-only로 본다.

## 7. 필요한 API gap

이 문서는 아래 gap을 예상한다.

1. 입력 배치 업로드 API
2. 업로드 검증 요약 API
3. 검증 통과 입력을 기준으로 run을 만드는 API
4. 결과 페이지에서 현재 run과 item을 묶어 읽는 요약 API

이 gap은 UI shell을 만든 뒤 필요한 범위만 다시 확정한다.

## 8. 현재 코드와의 관계

1. 현재 admin settlement는 그룹 라우트 분리까지는 됐다.
2. 하지만 `엑셀 업로드 중심`, `검증 요약`, `문맥 유지`, `실행 handoff`는 아직 current UI에 완전히 반영되지 않았다.
3. 따라서 이 문서는 다음 정산 UI 정리의 기준 문서다.
