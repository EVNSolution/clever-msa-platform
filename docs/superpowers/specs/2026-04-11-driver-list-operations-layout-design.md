# Driver List Operations Layout Design

## Purpose

이 문서는 `front-web-console`의 배송원 목록 화면을, 현재 승인된 운영 화면 스타일에 맞게 다시 정렬하는 기준을 고정한다.

이번 변경의 목표는 두 가지다.

- 배송원 목록을 `배차표 업로드`처럼 표 중심의 차분한 workbench 화면으로 재구성한다.
- `관리 > 메뉴관리`의 폼/상태 언어를 제한적으로 가져오되, 장식과 강조색은 최대한 줄인다.

## Reference Style

이번 화면은 아래 두 기존 화면을 기준으로 한다.

- `관리 > 메뉴관리`
- `배차표 업로드`

이 두 화면에서 추출한 스타일 규칙은 다음과 같다.

- 기본 표면은 흰 패널과 얇은 회색 보더를 사용한다.
- 강조색은 남발하지 않는다. 이 페이지에서는 연녹색 포인트를 쓰지 않는다.
- 필터는 단단한 폼 스타일만 유지하고, 별도 장식 컨테이너는 두지 않는다.
- 메타 정보는 작고 차분하게 노출한다.
- 표와 작업 대상이 먼저 보이게 하고, 요약칩이나 장식 요소가 앞서지 않게 한다.

## Decision 1: DriversPage Is Rebuilt As An Operations List Shell

`DriversPage`는 `대시보드형`이 아니라 `표 중심 운영 화면`으로 재구성한다.

### Rules

- `PageLayout.filters` 슬롯에 목록 필터 UI를 넣는다.
- 필터 영역은 별도 카드/컨테이너 없이 한 줄로 나열한다.
- 본문은 하나의 흰색 리스트 패널이 남은 높이를 우선 사용하도록 구성한다.
- 리스트 패널 내부는 `목록 본문 + 페이지네이션 푸터` 구조만 가진다.
- 데스크톱 기준 가로 스크롤을 기본 동작으로 삼지 않는다.
- 아주 좁은 화면에서만 마지막 안전장치로 테이블 영역 내부 스크롤을 허용한다.

### Why

- 사용자는 현재 배송원 목록이 “디자인이 별로”하다고 판단했고, 특히 장식성 높은 외피를 원하지 않았다.
- 기준 화면들은 모두 표 중심의 차분한 운영 화면이다.
- 현재 열 수는 많지 않아서, 가로 스크롤보다 밀도와 정보 정렬이 더 중요하다.

## Decision 2: Top Filters Are Rendered As Plain Controls

상단 필터는 `플릿 드롭다운`, `검색 입력`, `총 n명` 세 요소만 가진다.

### Rules

- 플릿 드롭다운은 `전체`와 현재 로드된 플릿 목록을 보여준다.
- 검색 대상은 `배송원 이름`, `원청 앱 사용자명(external_user_name)` 두 필드만 포함한다.
- `회사명`, `플릿명`, `계정 연결 사용자명`은 이번 검색 범위에 포함하지 않는다.
- 필터나 검색어가 바뀌면 현재 페이지는 자동으로 1페이지로 돌아간다.
- 검색 결과 수는 상단 오른쪽에 작은 메타 텍스트로 노출한다.
- 필터 영역 전체를 감싸는 추가 패널, 그라데이션, 요약 메타 블록은 두지 않는다.

### Non-Goal

- 이번 단계에서 서버 검색 API를 추가하지 않는다.
- 모든 필터와 검색은 현재 프런트가 이미 읽어 온 목록을 기준으로 처리한다.

## Decision 3: Repeated Summary UI Is Removed

이 페이지에서는 같은 정보를 여러 곳에 반복해서 노출하지 않는다.

### Rules

- 상단 요약칩을 두지 않는다.
- 선택된 플릿 칩을 두지 않는다.
- 검색어 칩을 두지 않는다.
- 범위 정보는 필요하면 푸터에서만 한 번 노출한다.
- 헤더와 푸터에 같은 범위 정보를 중복 표시하지 않는다.

### Why

- 사용자가 가장 강하게 지적한 부분이 중복 요소와 장식성이다.
- 기준 화면들은 메타 정보를 작게 쓰되, 같은 정보를 여러 층으로 반복하지 않는다.

## Decision 4: Pagination Is Quiet And Footer-Owned

페이지네이션은 리스트 하단 푸터에서만 관리하고, 시각적 존재감은 낮춘다.

### Rules

- 페이지 크기 옵션은 `10 / 25 / 50 / 전체`
- `전체`를 선택하면 필터 결과 전체를 한 페이지에 노출한다.
- 페이지 번호 버튼은 현재 필터 결과 기준으로 계산한다.
- 필터 결과가 0건이면 페이지 번호 대신 빈 상태를 보여준다.
- 푸터는 `노출 수`, `페이지 번호`, 필요 시 `범위 정보`만 가진다.
- 페이지네이션 버튼은 흰색 배경과 회색 보더를 기본으로 사용한다.
- 현재 페이지 강조도 연녹색이 아니라 회색 계열 대비만으로 처리한다.

## Decision 5: Row Density Is Increased Without Decorative Styling

행 높이는 줄이되, 클릭 가능성과 식별성은 유지한다.

### Rules

- 테이블은 계속 `compact` 계열을 사용하되, `DriversPage` 전용 밀도 스타일을 추가한다.
- 셀 패딩과 본문 글자 크기를 줄인다.
- 열 폭은 현재 화면에 맞춰 다시 배분한다.
  - 이름
  - 원청 앱 사용자명
  - 회사
  - 플릿
  - 계정 연결
- 긴 값은 말줄임 처리하되, 핵심 컬럼은 지나치게 잘리지 않게 우선순위를 둔다.
- 계정 연결 컬럼은 지금처럼 연결된 사용자 이름만 보여주고, 미연결이면 비워 둔다.
- 행 hover/focus 효과는 아주 약한 회색 계열만 쓴다.
- 그라데이션, 강한 그림자, 액센트 배경색은 쓰지 않는다.

## Data Flow

이번 단계에서는 데이터 흐름을 단순하게 유지한다.

- 초기 로드에서 `drivers`, `driverAccountLinks`, `companies`, `fleets`를 병렬로 읽는다.
- 화면 상태로 `selectedFleetId`, `searchTerm`, `pageSize`, `currentPage`를 가진다.
- 화면에서 파생 데이터로 `filteredDrivers`, `pagedDrivers`, `totalPages`를 계산한다.
- `driverAccountLinks`는 목록 표시용으로만 사용하고, 연결/해제 조작은 계속 상세 화면에 둔다.

## Layout Structure

- 상단: 제목/설명
- 그 아래: `플릿 드롭다운`, `검색 입력`, `총 n명`
- 본문: 흰색 리스트 패널 1개
- 하단: `노출 수`, `페이지 번호`, 필요 시 범위 정보 1회

## Error And Empty State

- 로드 실패 시 기존 `error-banner`를 그대로 사용한다.
- 필터 결과가 없으면 표 대신 `조건에 맞는 배송원이 없습니다.`를 리스트 본문 안에 표시한다.
- 데이터가 1페이지 이하라도 푸터 레이아웃은 유지하되, 페이지 번호는 최소 표현으로 줄인다.

## Test Strategy

`src/pages/DriversPage.test.tsx`에 아래 케이스를 추가하거나 갱신한다.

- 플릿 드롭다운이 목록을 올바르게 좁히는지
- 이름 검색이 동작하는지
- 원청 앱 사용자명 검색이 동작하는지
- 페이지 크기 `10 / 25 / 50 / 전체`가 동작하는지
- 페이지 이동 시 해당 페이지의 행만 보이는지
- 중복 요약칩이 렌더되지 않는지
- 기존 상세 이동 경로가 유지되는지
- 계정 연결 사용자 이름 표시가 유지되는지

## Acceptance Criteria

- 배송원 목록 행 높이가 기존보다 작아진다.
- 상단 필터는 별도 컨테이너 없이 한 줄 제어부로 보인다.
- 상단 요약칩, 검색어칩, 선택 플릿칩 같은 중복 UI가 없다.
- 목록 컨테이너가 본문 높이를 자연스럽게 사용한다.
- 하단에서 노출 수와 페이지 번호를 조작할 수 있다.
- `전체` 옵션이 실제로 동작한다.
- 계정 연결 조작은 목록에 없고, 연결된 사용자 이름만 표시된다.
- 연녹색 강조 없이도 화면이 차분한 운영 화면으로 보인다.
- 이번 변경은 `front-web-console` 프런트 단독 변경으로 끝난다.
