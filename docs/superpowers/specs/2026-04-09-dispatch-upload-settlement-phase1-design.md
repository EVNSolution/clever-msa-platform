# Dispatch Upload Settlement Phase 1 Design

## Purpose

이 문서는 참고 서비스의 `배차표 업로드 -> 인력 확정 -> 정산 생성` 운영 흐름을 CLEVER에 도입할 때의 1차 경계를 고정한다.

이번 단계의 목표는 아래 두 가지다.

- 배차표 업로드로 당일 배차 인력을 명확하게 확정한다.
- 정산은 확정된 인력과 박스 수를 읽어 계산한다.

이번 단계의 목표가 아닌 것은 아래다.

- 참고 서비스의 `팀 중심` 데이터 모델 복제
- 권역 정본 UUID 강제 매핑
- 전역 정산 설정을 다시 회사/플릿별 세금 설정으로 되돌리기

## Primary Decision

Phase 1의 기준선은 아래와 같다.

1. `배차`가 업로드를 소유한다.
2. `정산`은 배차에서 확정된 인력과 물량을 소비한다.
3. `세금/공제/보험`은 전역 정산 설정으로 유지한다.
4. `박스당 단가/특근비`는 회사·플릿별 단가표로 분리한다.

즉 업로드 파일은 정산 입력 화면의 수기 대체 수단이 아니라, `당일 배차 인력 확정용 source`다.

## Reference Interpretation

참고 서비스와 업로드 원본 파일을 기준으로 보면, 엑셀의 모든 컬럼이 정산 계산에 직접 쓰이지는 않는다.

실제 의미는 아래와 같이 해석한다.

- `배송매니저 이름`
  - 당일 투입 배송원 식별용 값
  - CLEVER 배송원 정본의 `external_user_name`과 매칭한다.
- `소분류 권역`, `세분류 권역`
  - 상하 관계를 가진 운영 참고값
  - phase 1에서는 문자열과 계층만 저장한다.
- `박스 수`
  - 정산 계산의 직접 근거값
- `가구 수`
  - 작업량/배송시간 예측용 근거값
  - 정산 계산의 직접 근거값으로 쓰지 않는다.

## Agreed Phase 1 Flow

Phase 1 운영 흐름은 아래 순서로 고정한다.

1. `배차표 업로드`
2. `당일 인력 매칭`
3. `용차/특근 보정`
4. `배차 확정`
5. `정산 기준 적용`
6. `정산 생성`
7. `정산 조회`

핵심 원칙은 아래 두 줄이다.

- 배차는 누가 실제로 일했는지를 확정한다.
- 정산은 그 확정 결과에 단가와 세금 규칙을 적용한다.

## Driver Matching Decision

업로드의 `배송매니저 이름`은 우리 인증 계정과 연결하지 않는다.

이 값은 원청사 배송원 앱 계정명을 의미하므로, `DriverProfile`의 새 속성 `external_user_name`에 저장한다.

### 이유

- `account_name`은 우리 로그인 계정으로 오해될 수 있다.
- `driver account`, `identity`, `session`과 같은 현재 인증 모델과 섞이면 의미가 흐려진다.
- 실제 운영 의미는 `원청사가 내려주는 외부 배송원 사용자명`이다.

### Phase 1 규칙

- 업로드 `배송매니저 이름 -> driver.external_user_name` 단일 매칭
- 인증 계정, 배송원 계정 링크와는 분리
- 매칭 실패 row는 배차 확정 전 검토 대상으로 남긴다.

## Region Handling Decision

권역은 phase 1에서 정산 키로 쓰지 않는다.

저장 원칙은 아래와 같다.

- `소분류 권역` 문자열 저장
- `세분류 권역` 문자열 저장
- 소분류 -> 세분류 상하 관계 유지
- 정본 `Region.region_id` 강제 매핑은 하지 않음

즉 phase 1의 권역은 정산 계산용 마스터 키가 아니라, 운영 검토용 문맥 데이터다.

## Settlement Inputs Decision

현재 `정산 입력` 화면은 delivery record와 daily snapshot을 직접 생성/수정하는 수기 CRUD 성격이 강하다.

Phase 1에서는 이 역할을 바꾼다.

- `배차`에서 업로드와 인력 확정이 먼저 일어난다.
- `정산 입력`은 업로드 결과로 만들어진 정산 대상 snapshot을 검토하는 화면이 된다.
- 예외 보정은 허용하지만, 원천 row 입력의 주 진입점은 아니다.

즉 `정산 입력`은 `record 작성 화면`이 아니라 `배차 결과 검토 화면`에 가까워져야 한다.

## Pricing Model Decision

정산 기준을 한 덩어리로 두지 않고 두 층으로 나눈다.

### 1. 전역 정산 설정

`service-settlement-registry`의 전역 설정으로 유지하는 대상:

- 세금
- 공제
- 보험
- 기타 전사 공통 계산 규칙

### 2. 회사·플릿별 단가표

별도 단가표로 분리하는 대상:

- 박스당 수신단가
- 박스당 지급단가
- 특근비

이렇게 나누는 이유는 아래와 같다.

- 1차 전역 방어를 유지할 수 있다.
- 실제 운영 단가는 회사/플릿 문맥을 가져야 한다.
- 세금/보험과 운영 단가를 같은 저장소 개념으로 섞지 않을 수 있다.

이 결정은 `GlobalSettlementConfig`를 되돌리는 것이 아니다.

- 전역 정산 설정은 계속 singleton으로 유지한다.
- 회사·플릿별 단가표는 별도 모델/계약로 분리한다.
- 즉 `세금/공제/보험`과 `단가/특근비`는 다른 저장소 개념으로 본다.

## Service Ownership

Phase 1 기준 서비스 소유 경계는 아래와 같다.

### `service-dispatch-registry`

- 배차표 업로드 배치
- 업로드 row 저장
- `external_user_name` 기반 배송원 매칭 결과
- 소분류/세분류 권역 문자열 저장
- 용차/특근 보정
- 당일 인력 확정 상태

업로드 저장 문맥은 phase 1에서 `company + fleet + dispatch_date` scope를 직접 가진다.

- `dispatch_plan` 연결은 future linkage이며 phase 1 MVP의 필수 조건이 아니다.
- 같은 `company + fleet + dispatch_date` 문맥에서 upload batch를 해석한다.
- 업로드는 독립 정산 입력 정본이 아니라 배차 정본의 보조 source다.

### `service-driver-profile`

- `DriverProfile.external_user_name` 정본

### `service-settlement-registry`

- 전역 정산 설정
- 회사·플릿별 단가표

### `service-delivery-record`

- 배차 확정 결과에서 정산용 snapshot 생성

### `service-settlement-payroll`

- 확정 snapshot + 단가표 + 전역 정산 설정을 읽어 정산 생성

### `service-settlement-operations-view`

- 정산 결과 read 모델

## Frontend IA Decision

Phase 1의 화면 책임은 아래처럼 나눈다.

### 배차

- 배차 계획
- 배차표 업로드
- 업로드 결과 검토
- 용차/특근 보정
- 당일 인력 확정

### 정산 처리

- 정산 기준
- 정산 입력 검토
- 정산 실행
- 정산 결과

### 정산 조회

- 운영 요약
- 실행 결과
- 기사별 결과

즉 `배차`는 실제 작업자 확정, `정산`은 계산과 결과 관리로 역할을 분리한다.

phase 1 MVP에서는 `배차 계획`과 `배차표 업로드`를 별도 entry로 둔다.

- `배차 계획`은 `dispatch_plan` 정본과 예상 물량 운영을 소유
- `배차표 업로드`는 `dispatchPlan` 없이도 업로드/확정/정산 준비를 시작
- 이후 phase에서 두 흐름을 같은 scope 안에서 연결한다

### Upload Date Detection Rule

배차표 업로드는 파일명에서 `dispatch_date`를 보조 감지할 수 있다.

- 지원 패턴은 `YYYY-MM-DD`, `YYYY_MM_DD`, `YYYYMMDD`
- 감지된 날짜는 즉시 적용하지 않고 사용자 확인 한 번을 거친다
- 지원 패턴이 아니거나 애매하면 사용자가 날짜를 직접 선택한다

## Phase 1 Non-Goals

이번 단계에서 하지 않는 것:

- 참고 서비스의 `팀` 엔티티 복제
- 권역 정본 강제 매핑
- 업로드 row를 정산 write에 직접 연결
- 세금/보험의 회사·플릿별 분기
- 외부 배송원 사용자명 alias 다건 구조

Phase 1에서는 `external_user_name` 단일 필드와 수동 검토 흐름만으로도 충분하다고 본다.

## Acceptance Criteria

Phase 1 완료 기준은 아래와 같다.

- 배차표 업로드 후 `배송매니저 이름`으로 당일 배송원 매칭이 가능하다.
- `박스 수` 기준으로 정산용 snapshot이 만들어진다.
- `가구 수`는 정산 계산에 직접 쓰이지 않는다.
- 권역은 문자열/계층 정보로 조회 가능하다.
- 용차/특근 보정 후 당일 인력 확정이 가능하다.
- 정산은 확정된 인력과 snapshot을 읽어 계산된다.
- 세금/공제/보험은 전역 설정에서 유지된다.
- 박스당 단가/특근비는 회사·플릿별 단가표에서 관리된다.
