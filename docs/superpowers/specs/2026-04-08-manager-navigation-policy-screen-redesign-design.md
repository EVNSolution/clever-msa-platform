# Manager Navigation Policy Screen Redesign

## Purpose

이 문서는 `/admin/menu-policy` 화면을 전면 재설계하기 위한 상위 설계를 고정한다.

현재 화면은 전역 관리자 네비게이션 정책을 편집할 수는 있지만, 아래 문제가 크다.

- 역할 선택과 정책 편집이 한 덩어리로 붙어 있어 정보 밀도가 과하다.
- 메뉴 설명 없이 체크박스만 보여서 정책 의미가 불분명하다.
- 전역 정책, 회사 override, 정책 관리용 메타 메뉴가 한 수준에서 섞여 있다.
- 저장 전후 차이와 현재 효과가 보이지 않는다.
- 정책 화면인데 실제 사이드바 결과를 예측하기 어렵다.

목표는 단순히 UI를 꾸미는 것이 아니라, 정책 편집을 `역할별 정책 워크벤치`로 바꾸는 것이다.

## Primary Decision

`/admin/menu-policy`는 단일 폼이 아니라 아래 3영역 워크벤치로 재구성한다.

1. 역할 목록 레일
- 한 번에 한 역할만 편집한다.
- 드롭다운 대신 카드/리스트 선택 구조로 바꾼다.
- 각 역할의 현재 정책 상태를 요약해 보여준다.

2. 정책 편집 패널
- 메뉴 그룹별로 정리된 정책 편집 본체다.
- 각 메뉴 항목은 `이름 + 설명 + 현재 상태 + 변경 여부`를 함께 보여준다.
- 저장은 전체 정책이 아니라 현재 선택 역할 단위로 수행한다.

3. 사이드바 미리보기 패널
- 지금 선택한 역할에 대해 실제로 보일 사이드바를 오른쪽에 즉시 렌더한다.
- 저장 전에도 편집 상태를 반영해 사용자가 결과를 이해할 수 있게 한다.

## UX Decisions

### 1. 역할 선택은 드롭다운이 아니라 역할 레일

현재 드롭다운은 역할 간 비교와 상태 파악이 어렵다.

새 화면에서는 역할 레일을 둔다.

- `회사 전체 관리자`
- `차량 관리자`
- `정산 관리자`
- `플릿 관리자`

각 항목에는 아래 요약을 같이 둔다.

- 허용 메뉴 개수
- 기본값 대비 변경 여부
- 저장되지 않은 변경 유무

### 2. 메뉴는 체크박스 나열이 아니라 정책 항목 카드

각 메뉴 항목은 최소 아래 정보를 가진다.

- 표시 이름
- 내부 key
- 짧은 설명
- 그룹명
- 현재 상태
  - 허용
  - 기본값 상속
  - 숨김

예시:

- `회사`
  - key: `companies`
  - 설명: 회사 상세, 플릿, 계정 연결, 조직 운영 진입점
- `차량`
  - key: `vehicles`
  - 설명: 차량 등록, 상세, 운영자 연결 관리

### 3. 메타 정책 메뉴는 일반 메뉴와 분리

아래 항목은 일반 운영 메뉴와 같은 그룹에 놓지 않는다.

- `manager_navigation_policy`
- `company_navigation_policy`

이 둘은 정책 관리 전용 항목이므로, 전역 정책 편집 대상 목록의 별도 `정책 관리` 섹션으로 분리하거나 1차에서는 아예 편집 대상에서 제외한다.

1차 결정:
- `/admin/menu-policy`의 전역 정책 편집에서는 위 두 key를 기본적으로 숨긴다.
- 시스템 관리자 기본 정책은 코드상 항상 허용으로 유지한다.

### 4. 저장 단위는 현재 역할만

현재 전체 정책 PUT 방식은 사용자가 무엇을 저장하는지 감이 없다.

새 화면에서는 현재 선택 역할만 저장한다.

백엔드가 내부적으로 전체 리스트 payload를 유지하더라도, 프론트 UX는 역할 단위 저장으로 고정한다.

### 5. 미리보기는 실시간 반영

오른쪽 패널에서 현재 편집 상태를 즉시 반영한 사이드바를 보여준다.

이 미리보기는 아래 용도를 가진다.

- 저장 전 검토
- 역할별 차이 확인
- 정책 변경이 실제 사용자 경험에 미치는 영향 확인

## Information Architecture

정책 편집 패널은 아래 그룹 순서로 정리한다.

1. 기본
- 대시보드
- 내 계정

2. 운영
- 계정 요청
- 공지
- 지원
- 알림

3. 조직/배차
- 회사
- 권역
- 배차

4. 현장
- 배송원
- 인사문서
- 차량
- 차량 배정
- 정산

정책 관리 항목은 1차에서 편집 대상에서 제외한다.

## State Model

화면에서 다루는 상태는 아래처럼 구분한다.

- persisted policy
- working draft policy
- inherited/global default policy
- dirty state

즉 사용자는 아래를 명확히 봐야 한다.

- 지금 저장된 정책이 무엇인지
- 내가 무엇을 바꿨는지
- 이 역할의 기본값은 무엇인지
- 아직 저장하지 않은 변경이 있는지

## Copy Decisions

문구는 정책 의미가 분명하게 바뀌어야 한다.

현재 화면의 추상적 문구 대신 아래처럼 간다.

- 제목: `전역 관리자 메뉴 정책`
- 설명: `관리자 유형별로 기본 사이드바 메뉴를 정의합니다. 회사별 정책이 있으면 그 값이 우선 적용됩니다.`
- 저장 버튼: `현재 역할 저장`
- 복원 버튼: `기본 정책으로 되돌리기`
- 변경 배지: `변경됨`
- 상속 배지: `기본값 사용`

## Backend Contract Decision

이번 재설계는 우선 프론트 중심이다.

백엔드 API shape는 크게 바꾸지 않아도 된다. 다만 아래가 필요하다.

- 현재 역할 단위 저장이 가능하도록 프론트 payload 구성 단순화
- 정책 응답에 `source`, `allowed_nav_keys`, `role_type`은 유지
- 필요 시 각 nav item의 메타데이터는 프론트 로컬 정의로 둔다

즉 1차에서는 백엔드 계약을 최소한으로 건드리고, 프론트 정보 구조를 다시 세운다.

## Relationship With Company Override

`/company/menu-policy`도 같은 정보 구조를 따라야 한다.

다만 현재 우선순위는 전역 정책 화면이다.

전역 화면을 먼저 바로잡은 뒤, 회사 override 화면도 같은 워크벤치 구조로 맞춘다.

## Final Decision Summary

이번 재설계의 최종 결정은 아래다.

1. `/admin/menu-policy`는 3영역 워크벤치로 재설계한다.
2. 역할 선택은 드롭다운이 아니라 역할 레일로 바꾼다.
3. 메뉴는 단순 체크박스가 아니라 설명 있는 정책 항목 카드로 보여준다.
4. 정책 관리용 메타 메뉴는 일반 운영 메뉴와 분리한다.
5. 저장 단위는 현재 역할 단위로 단순화한다.
6. 오른쪽에 실시간 사이드바 미리보기를 추가한다.
7. 이 구조를 이후 `/company/menu-policy`에도 재사용한다.
