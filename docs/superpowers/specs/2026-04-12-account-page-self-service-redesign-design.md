# 내 계정 페이지 self-service 재설계

## 목적

- `내 계정` 페이지를 운영 콘솔 관리 화면처럼 보이지 않게 줄인다.
- `내 정보 관리` 문맥만 남기고, 카드 순서와 폭을 읽기 좋은 흐름으로 재배치한다.
- 현재 기능은 유지하되, 산만한 좌우 확장과 역할 혼합 표현을 제거한다.

## 현재 문제

- 페이지가 좌우로 넓게 퍼져 개인 설정 화면보다 운영 대시보드처럼 보인다.
- `현재 웹 권한`, `self-service`, `요청 생성과 회사 변경`처럼 제목 톤이 제각각이라 페이지 목적이 흐린다.
- 같은 맥락 설명이 여러 카드에서 반복돼 시선이 분산된다.
- `내 요청` 이력은 큰 카드 반복이라 self-service 페이지 리듬과 맞지 않는다.

## 범위

- 대상 repo: `development/front-web-console`
- 대상 화면: `src/pages/AccountPage.tsx`
- 이 작업은 UI/UX 재구성만 다룬다.
- API 계약, 데이터 구조, 기능 범위는 바꾸지 않는다.

## 유지할 기능

- 기본 정보 조회/수정
- 필수 동의 상태 표시와 철회
- 로그인 수단 조회/추가/삭제
- 내 요청 생성/취소
- 비밀번호 변경
- 현재 권한/현재 회사 요약

## 비포함

- 요청 종류/동의/로그인 수단의 정책 변경
- 새 API 추가
- 내 계정 범위를 넘는 관리자 운영 기능 추가

## 레이아웃 원칙

- 헤더는 기존 `PageLayout` 템플릿을 유지한다.
- 헤더 아래 첫 패널을 다시 헤더처럼 쓰지 않는다.
- 전체 콘텐츠 폭은 제한해 좌우 끝까지 퍼지지 않게 한다.
- 데스크톱은 `요약 1열 + 본문 2열 + 비밀번호 1열` 구조로 간다.
- 중간 폭 이하에서는 본문 2열을 1열로 접는다.
- 모바일에서는 전부 1열로 떨어뜨린다.

## 정보 구조

### 1. 현재 접근 요약

- 헤더 바로 아래의 작은 요약 카드 하나만 둔다.
- 남길 정보는 `현재 권한`, `현재 회사` 두 개다.
- 설명 문구는 최소화한다.
- 이 카드는 “현재 웹 권한” 같은 운영식 제목 대신 `현재 접근` 또는 `현재 정보` 톤으로 정리한다.

### 2. 본문 2열

- 왼쪽 열
  - `기본 정보`
  - `필수 동의`
- 오른쪽 열
  - `로그인 수단`
  - `내 요청`

이 순서는 “개인 정보 확인 -> 동의 상태 확인 -> 로그인 수단 관리 -> 요청 확인” 흐름을 기준으로 한다.

### 3. 최하단 단독 카드

- `비밀번호 변경`
- 현재 비밀번호와 새 비밀번호를 다루는 민감 액션이라 최하단 분리 카드로 둔다.
- 본문 중간에 섞지 않는다.

## 카드별 설계

### 기본 정보

- 제목: `기본 정보`
- 필드: 이름, 생년월일
- 액션: `저장`
- 카드 footer 또는 하단 action row에 버튼을 둔다.

### 필수 동의

- 제목: `필수 동의`
- 항목: 개인정보처리, 위치기반
- 각 항목은 현재 상태를 짧게 보여준다.
- 철회 버튼 2개는 같은 행 baseline이 맞게 둔다.
- 불필요한 보조 설명은 넣지 않는다.

### 로그인 수단

- 제목: `로그인 수단`
- 현재 로그인 수단 목록을 먼저 보여준다.
- 추가 영역은 아래에 compact form으로 둔다.
- `추가 방식`, 입력 필드, `추가` 버튼이 한 문맥으로 정렬돼야 한다.
- 마지막 수단 삭제용 현재 비밀번호 입력도 이 카드 안에 포함하되, 목록과 분리된 보조 row로 둔다.

### 내 요청

- 제목: `내 요청`
- 상단은 compact 요청 생성 폼
  - 회사 선택
  - 요청 종류
  - 회사 변경 요청 체크
  - 제출 버튼
- 하단은 요청 이력
  - 큰 카드 반복 대신 compact row list로 줄인다.
  - 각 row는 `요청명 / 회사 / 상태 / 요청일 / 요청 취소` 수준으로만 보여준다.
- 안내 문구는 한 곳에만 두고 반복하지 않는다.

### 비밀번호 변경

- 제목: `비밀번호 변경`
- 위치: 페이지 최하단
- 필드: 현재 비밀번호, 새 비밀번호
- 액션: `저장`
- 다른 카드와 분리해 민감 액션임을 분명히 한다.

## 표현 규칙

- 카드 제목은 전부 `내 계정` 관점 용어로 통일한다.
- `self-service`, `운영`, `요청 생성과 회사 변경` 같은 혼합 표현은 제거한다.
- 같은 설명을 헤더/카드/목록에 반복하지 않는다.
- 버튼, 드롭다운, 입력창은 같은 row에서 baseline이 맞아야 한다.
- 페이지 전체가 운영 테이블 화면처럼 보이지 않도록 표보다 카드 중심 리듬을 쓴다.

## 반응형

- 데스크톱
  - 폭 제한 컨테이너
  - 본문 2열
- 태블릿/중간 폭
  - 본문 1열 전환
  - 카드 내부 액션 row 줄바꿈 허용
- 모바일
  - 모든 카드 1열
  - 버튼/입력 높이 정렬 유지
  - height가 낮은 환경에서도 카드 내부 액션이 밀리지 않게 확인

## 테스트와 검증

- `AccountPage.test.tsx`를 현재 구조 기준으로 업데이트한다.
- 검증 포인트
  - `현재 접근` 요약이 작게 보이는지
  - 본문이 2열에서 1열로 자연스럽게 접히는지
  - `내 요청` 이력이 compact row list로 보이는지
  - `비밀번호 변경`이 최하단에 단독 카드로 남는지
  - 버튼, select, input baseline이 맞는지

## 구현 메모

- 전역 CSS를 건드리지 않는다.
- `AccountPage` 전용 클래스 축으로 묶는다.
- 필요하면 `PageLayout`의 `contentClassName` 또는 명시적 wrapper 클래스만 사용한다.
