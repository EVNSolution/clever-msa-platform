# 02. Target Roles and Join Risks

## 문서 목적

이 문서는 MSA 후보 서비스를 실제 구조도로 올리기 전에 각 서비스의 역할, 쓰기 권한, 판단 권한, 읽기 의존을 먼저 고정하는 기준 문서다.

이번 단계에서 먼저 확정하려는 것은 아래 네 가지다.

1. 각 서비스가 무엇의 주인인지
2. 각 서비스가 무엇까지 최종 판단할 수 있는지
3. 다른 서비스 데이터를 어디까지 읽기만 해야 하는지
4. 어떤 화면과 업무 흐름에서 과도 조인이 발생할지를 미리 어디서 막아야 하는지

## 기본 원칙

### 1. 서비스는 자기 데이터만 쓴다

- 한 서비스의 테이블이나 문서를 다른 서비스가 직접 수정하지 않는다.
- 서비스 간 참조는 외부 식별자와 이벤트, API, 읽기 모델로만 연결한다.

### 2. 상태 판단 권한도 한 곳에 둔다

- 데이터를 저장하는 서비스와 최종 상태를 판단하는 서비스가 다르면 다시 결합된다.
- 예를 들어 계정 잠금은 Identity Access가 판단하고, 배송원 재직 상태는 Driver Profile HR이 판단한다.

### 3. 운영 화면 조합은 읽기 모델로 푼다

- 여러 서비스 데이터를 한 화면에 보여줘야 하는 요구는 자연스럽다.
- 이 요구를 서비스 간 실시간 다중 호출과 런타임 조인으로 풀지 말고, 별도 읽기 모델이나 프로젝션으로 푼다.

### 4. 논리 경계와 물리 배치는 분리한다

- 논리 서비스는 세밀하게 나눌 수 있다.
- 하지만 초기 물리 MSA는 과도 조인을 줄이기 위해 더 적은 수의 배포 단위로 시작하는 편이 안전하다.

## 논리 서비스 구조

### 1. Identity Access

| 항목 | 내용 |
|---|---|
| 핵심 역할 | 로그인 가능한 주체 관리, 자격 검증, 토큰과 세션 관리 |
| 소유 데이터 | 계정, 자격, 계정 링크, 로그인 시도, 잠금 상태, 마지막 로그인 |
| 쓰기 권한 | 계정 생성과 비활성화, 비밀번호와 인증수단 관리, 토큰 발급과 폐기 |
| 판단 권한 | 로그인 가능 여부, 자격 유효 여부, 잠금 정책 |
| 읽기 의존 | Driver, Vehicle, Ops 대상과의 연결 상태 확인 |
| 가져가면 안 되는 것 | 배송원 계약 상태, 차량 운영 상태, 운영 조직 권한 상세 |

### 2. Organization Master

| 항목 | 내용 |
|---|---|
| 핵심 역할 | 회사, 조직, 플릿, 부서, 포지션, 기준 조직 구조 관리 |
| 소유 데이터 | 회사, 법인, 조직 트리, 플릿, 부서, 기준 분류 |
| 쓰기 권한 | 조직 생성과 변경, 소속 구조 변경 |
| 판단 권한 | 조직 소속의 정본, 플릿과 부서의 활성 여부 |
| 읽기 의존 | 없음 또는 최소화 |
| 가져가면 안 되는 것 | 배송원 개인 정보, 차량 상세, 계정 자격 |

### 3. Driver Profile HR

| 항목 | 내용 |
|---|---|
| 핵심 역할 | 배송원 프로필, 인사성 상태, 업무 연락처, 입퇴사와 소속 정보 관리 |
| 소유 데이터 | 배송원 프로필, 업무 연락처, 계약 유형, 근무 유형, 회사 소속 참조, 플릿 소속 참조, 연차, 입퇴사 정보 |
| 쓰기 권한 | 배송원 생성과 수정, 재직 상태 변경, 소속과 인사 정보 변경 |
| 판단 권한 | 재직 여부, 기사 프로필 완성도, 업무상 활성 상태 |
| 읽기 의존 | Organization Master의 회사와 플릿 마스터, Identity Access 요약 상태 |
| 가져가면 안 되는 것 | 로그인 잠금, 토큰 상태, 차량 텔레메트리 |

### 4. Vehicle Asset

|---|---|

## 서비스별 권한 해석 규칙

### Identity Access만 결정할 수 있는 것

- 로그인 허용 여부
- 계정 잠금 여부
- 자격 만료와 재설정 여부

### Driver Profile HR만 결정할 수 있는 것

- 배송원 재직 여부
- 배송원 업무 연락처
- 배송원 계약 상태와 현재 회사/플릿 소속 참조

### Vehicle Asset만 결정할 수 있는 것

- 차량 사용 가능 여부
- 차량 자산 속성
- 차량 정비와 사고 상태

### Driver Vehicle Assignment만 결정할 수 있는 것

- 오늘 근무 배정
- 기사-차량 현재 매칭 상태
- 배정과 반납 workflow 상태

### Delivery Execution만 결정할 수 있는 것

- 배송 진행 상태
- 배송지 완료 여부
- 배송 수행 로그

### Settlement Payroll만 결정할 수 있는 것

- 정산 계산 결과
- 공제와 인센티브 반영 결과
- 정산 식별자 매핑

## 권장 물리 배치

논리 서비스는 위처럼 분리하되, 초기 물리 배치는 아래처럼 더 적은 수로 시작하는 편이 안전하다.

| 물리 배치 | 포함 논리 서비스 | 이유 |
|---|---|---|
| Access Platform | Identity Access | 인증과 세션은 독립성이 가장 높고 다른 도메인과 쓰기 결합이 낮다 |
| People Platform | Organization Master, Driver Profile HR, Talent Acquisition | 사람과 조직 축은 조회 결합이 강하고 채용 전환도 잦다 |
| Vehicle Platform | Vehicle Asset, Terminal Ops | 차량과 단말은 운영 화면에서 거의 항상 함께 보인다 |
| Operations Platform | Driver Vehicle Assignment, Delivery Execution, Driver Location, Route Knowledge and Region | 배차와 배송 실행, 위치, 경로 지식은 관제 흐름에서 함께 움직인다 |
| Finance Platform | Settlement Payroll, Approval Workflow | 정산과 결재는 승인 흐름과 배치성 계산이 강하게 연결된다 |
| Support Platform | Communication Support | 지원성 메시지와 티켓은 코어 쓰기 모델의 정본을 소유하지 않는다 |
| Telemetry Platform | Telemetry Hub | 시계열 수집량과 접근 패턴이 달라 별도 배치가 낫다 |

이 구조는 논리 서비스 14개를 유지하면서도 초기 물리 서비스 수를 7개 수준으로 낮춘다.

## 과도 조인 위험 지점

### 1. 배송원 상세 운영 화면

| 항목 | 내용 |
|---|---|
| 필요한 데이터 | 계정 상태, 배송원 프로필, 소속 조직, 오늘 스케줄, 현재 차량, 최근 정산 |
| 엮이는 서비스 | Identity Access, Driver Profile HR, Organization Master, Driver Vehicle Assignment, Vehicle Asset, Settlement Payroll |
| 위험도 | 매우 높음 |
| 대응 | Driver 360 읽기 모델을 별도 운영하고 화면은 읽기 모델만 조회 |

### 2. 차량/터미널 운영 화면

| 항목 | 내용 |
|---|---|
| 필요한 데이터 | 차량 기본 정보, 단말 상태, 현재 기사, 인수인계 상태, 최신 텔레메트리 |
| 엮이는 서비스 | Vehicle Asset, Terminal Ops, Driver Vehicle Assignment, Driver Profile HR, Telemetry Hub |
| 위험도 | 매우 높음 |
| 대응 | Vehicle Ops 읽기 모델과 차량 단위 타임라인 프로젝션 구성 |

### 3. 일일 배차/배송 관제 화면

## 읽기 모델이 반드시 필요한 화면

1. Driver 360
2. Vehicle Ops
3. Dispatch Control
4. Settlement Review Board
5. Approval Inbox

이 다섯 화면은 서비스 분리 이후에도 단일 서비스 조회만으로는 충분하지 않다.

## 서비스 간 호출 규칙

1. 쓰기 요청 중에는 다른 서비스 DB를 직접 조회하거나 조인하지 않는다.
2. 사용자 화면에서 동기 호출이 세 서비스를 넘기기 시작하면 읽기 모델 후보로 본다.
3. 결재와 알림에는 원본 데이터 전문이 아니라 요약 스냅샷을 전달한다.
4. 서비스 간 참조는 내부 PK가 아니라 외부 식별자로 연결한다.
5. 조직, 기사, 차량, 계정 상태처럼 자주 보이는 조합은 처음부터 프로젝션 설계를 함께 잡는다.
