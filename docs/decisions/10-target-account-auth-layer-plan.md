# 10. Target Account Auth

## 문서 목적

이 문서는 현재 서버의 User-Document 구조를 기준으로 Account/Auth 레이어를 별도 도메인으로 분리하기 위한 첫 계획 초안이다.

이번 단계의 목표는 구현이 아니라 아래 세 가지를 먼저 고정하는 것이다.

1. Account/Auth가 무엇을 소유하는지
2. Driver, Vehicle, Ops, Settlement가 무엇을 계속 소유하는지
3. 현재 결합 구조를 어떤 순서로 끊어낼지

## 현재 상태 요약

현행 구조는 인증 주체와 업무 주체가 강하게 결합되어 있다.

- User는 로그인 계정이지만 배송원 Document 연결, 차량 번호, 관리자 연락처, 회사, 플릿, 역할을 함께 가진다.
- Document는 배송원 프로필이지만 이름, 연락처, 계약, 차량, 조직, 정산 관련 성격까지 함께 가진다.
- 배송원 Document 생성 시 User가 자동 생성된다.
- 후보자 전환 과정에서도 Document와 User가 함께 생성된다.
- User 상태는 Document의 필수 정보와 계약 정보 완성도를 기준으로 계산된다.
- Document 화면의 탭 카운트와 반려 처리도 연결된 User 상태를 기준으로 동작한다.
- 드라이버 OTP 로그인은 사실상 전화번호를 User 식별자로 사용한다.
- 공유계정은 로그인 계정이 아니라 정산/기록 식별자에 가까운데도 Driver 유형 축에 섞여 있다.
- role 하나에 계정 종류, 업무 성격, 운영 권한 의미가 동시에 들어가 있다.

즉 현재 구조는 로그인, 승인, 인사, 차량 연결, 정산 식별이 완전히 분리되지 않은 상태다.

## 문제 정의

이 구조에서 가장 큰 문제는 계정 서비스가 얇지 않다는 점이다.

- 인증용 식별자와 업무용 연락처가 같은 축으로 취급된다.
- 배송원 승인 상태와 계정 로그인 가능 상태가 섞여 있다.
- 차량 계정과 사람 계정이 같은 role 체계 안에 놓여 있다.
- 운영 권한과 접속 채널 구분이 한 컬럼에 섞여 있다.
- 공유계정 같은 비로그인 식별자가 사람 계정 구조 안으로 들어온다.
- 운영 화면은 편하지만 쓰기 모델의 책임 경계는 계속 흐려진다.

결과적으로 어느 도메인에서 무엇이 SSOT인지 설명하기 어려워지고, 새 요구사항이 생길 때마다 User나 Document에 필드가 계속 추가되는 구조가 된다.

## 목표 상태

지향점은 유저 통합 서비스가 아니라 계정 서비스다.

Account/Auth 레이어는 오직 로그인 가능한 주체와 인증 절차만 다룬다.

- 누가 로그인 가능한가
- 어떤 방식으로 인증하는가
- 지금 로그인 가능한 상태인가
- 이 계정이 어떤 외부 도메인 엔티티와 연결되는가

반대로 아래 정보는 Account/Auth가 소유하지 않는다.

- 배송원 인사 정보
- 계약, 소속, 플릿, 근무 유형
- 차량 운영 상태와 차량 속성
- 운영자 조직 권한 상세
- 공유계정과 정산 식별자

## 서비스 경계 초안

### 1. Account Service

책임

- 로그인 가능한 주체의 등록
- 계정 종류 관리
- 계정 상태 관리
- 계정과 외부 도메인 엔티티의 연결 관리
- 인증 자격 메타데이터 보관

비책임

- 배송원 프로필 관리
- 차량 실체 관리
- 운영 조직 권한 해석
- 정산 대상 관리

### 2. Auth Service

책임

- 로그인 진입점 제공
- 자격 검증
- OTP, 비밀번호, 임시 비밀번호 등 인증 플로우 실행
- 토큰, 세션, 로그인 시도, 잠금 정책 관리
- 마지막 로그인 기록

비책임

- 프로필 조회
- 인사 승인
- 차량 배정
- 운영 권한 상세 계산

### 3. Driver Profile Service

책임

- 배송원 프로필
- 인사/계약/소속/플릿/근태/정산에 필요한 기준 정보
- 업무 연락처
- 입사/퇴사와 업무 상태

### 4. Vehicle Service

책임

- 차량 실체
- 차량 번호와 차대번호
- 단말 연결 상태
- 배정 상태와 운영 상태

### 5. Ops Staff Service

책임

- 운영자 프로필
- 소속 부서와 조직 구조
- 운영 권한 묶음
- 접근 가능한 조직 범위

### 6. Settlement Identity Service

책임

- 공유계정
- 정산용 식별자
- 실배송원과 분리된 정산 대상

## Account/Auth SSOT

| 영역 | Account/Auth가 SSOT인 것 | Account/Auth가 SSOT가 아닌 것 |
|---|---|---|
| 계정 | 계정 존재, 계정 종류, 계정 상태, 생성 시각, 마지막 로그인 | 배송원 이름, 차량 번호, 운영자 부서 |
| 인증 | 인증 방식, 자격 유형, 비밀번호 해시, OTP 검증 상태, 잠금 상태 | 근무 유형, 계약 상태, 차량 상태 |
| 연결 | 연결 대상 도메인 종류, 외부 엔티티 식별자, 연결 상태 | 연결 대상의 상세 속성 |
| 접근 | 로그인 허용 여부, 세션/토큰 유효성 | 세부 업무 권한, 조직 범위 |

## 핵심 모델 초안

### Account

공통 계정 엔티티는 최소 필드만 가진다.

- account_id
- account_type
- account_status
- created_at
- last_login_at

추천 account_type

- 운영 웹 계정
- 배송원 앱 계정
- 차량 앱 계정

추천 account_status

- 아직 활성화되지 않음
- 활성
- 잠김
- 비활성
- 폐기됨

이 상태는 인증과 접속 관점에서만 해석한다.

### Credential

자격은 계정과 분리하고 유형별로 나눈다.

운영 웹 자격

- 로그인 ID
- 비밀번호 해시

배송원 자격

- 전화번호
- 외부 인증 식별자 또는 카카오 식별 키

차량 자격

- 차대번호
- 임시 비밀번호 해시
- 초기 비밀번호 재설정 필요 여부

### Account Link

계정과 외부 도메인 엔티티를 연결하는 모델이다.

- account_id
- domain_type
- domain_entity_id
- link_status
- linked_at

예시

- 배송원 앱 계정은 Driver Profile 하나에 연결된다.
- 차량 앱 계정은 Vehicle 하나에 연결된다.
- 운영 웹 계정은 Ops Staff 하나에 연결된다.

## 축 분해 원칙

현행 role 하나에 섞여 있는 의미를 최소 네 축으로 분해한다.

| 축 | 질문 | 예시 |
|---|---|---|
| 계정 종류 | 무엇을 위한 계정인가 | 운영 웹, 배송원 앱, 차량 앱 |
| 인증 방식 | 어떻게 로그인하는가 | ID/PW, 전화번호 기반 인증, 차대번호/임시 비밀번호 |
| 도메인 역할 | 연결된 엔티티의 업무 성격은 무엇인가 | 정규 배송원, 용차 배송원, 차량, 운영자 |
| 권한 | 어떤 업무 권한을 가지는가 | 전체 관리자, 물류 관리자, 차량관리 사용자 |

이 네 축은 서로 독립적으로 설계해야 한다.

## 현재 구조에서 바로 끊어야 할 결합

1. Document 생성 시 User를 자동 생성하는 흐름
2. User 상태를 Document 완성도로 직접 계산하는 흐름
3. 전화번호를 계정 식별자와 프로필 연락처로 동시에 소유하는 구조
4. role 하나로 계정 종류와 운영 권한을 동시에 표현하는 구조
5. 공유계정을 Driver 유형 안에 넣는 구조

## 전환 전략

### Phase 0. 용어 고정

먼저 용어를 확정한다.

- User 대신 Account
- 배송원 인사 객체는 Driver Profile
- 운영자 객체는 Ops Staff
- 공유계정은 Settlement Identity

이 단계에서 기존 User와 Document를 그대로 두더라도 문서와 API 계약에서는 새 용어를 사용한다.

### Phase 1. 논리 경계 먼저 분리

물리 MSA 이전에 서버 내부 모듈 경계부터 분리한다.

- Account 모듈
- Auth 모듈
- Driver Profile 모듈
- Vehicle 모듈
- Ops Staff 모듈
- Settlement Identity 모듈

이 단계에서는 같은 저장소 안에 있어도 쓰기 책임이 어디에 있는지부터 끊어낸다.

### Phase 2. Account/Auth 정규 모델 도입

새 공통 모델을 먼저 만든다.

- Account
- Credential
- Account Link
- Login Attempt
- Session 또는 Token 발급 기록

기존 User는 바로 제거하지 않고 호환 계층으로 둔다.

### Phase 3. 드라이버 로그인 분리

가장 결합이 강한 배송원 축부터 분리한다.

- 드라이버 로그인 식별자는 Driver 자격 모델이 소유한다.
- Driver Profile의 업무 연락처는 Driver Profile 도메인이 계속 소유한다.
- Account는 Driver Profile과 링크만 가진다.
- 로그인 가능 여부는 Account 상태와 Auth 정책으로 판단한다.

### Phase 4. 운영 웹과 차량 계정 분리

운영 웹과 차량은 드라이버와 인증 방식이 다르므로 자격 모델을 독립시킨다.

- 운영 웹은 ID/PW 기반
- 차량은 차대번호 또는 차량 고유식별자 기반

이 단계에서 vehicle role 같은 혼합 표현을 걷어낸다.

### Phase 5. 공유계정 분리

공유계정은 Account 영역 밖으로 이동한다.

- 로그인하지 않는 식별자
- 정산과 기록을 위한 대상
- 실배송원과 별도 생명주기 보유

### Phase 6. 읽기 모델 구성

운영 화면 편의를 위해 별도 읽기 모델을 만든다.

예시 화면 조합

- 계정 상태 + 배송원 프로필 + 차량 배정
- 운영자 계정 상태 + 조직 권한
- 차량 계정 상태 + 차량 운영 상태

쓰기 모델은 분리하고, 운영 편의는 읽기 모델로 해결한다.

## 초기 구현 범위 제안

첫 단계에서는 물리 서비스 분리보다 아래를 먼저 완료하는 편이 안전하다.

1. 용어와 책임 정의서 확정
2. Account, Credential, Account Link 개념 모델 확정
3. role 분해안과 권한 모델 초안 확정
4. Driver, Vehicle, Ops, Settlement별 외부 식별자 규칙 확정
5. 운영 화면에 필요한 읽기 모델 목록 정리

이번 단계에서 하지 않을 것

- 즉시 데이터베이스 분리
- 즉시 토큰 체계 전면 교체
- 모든 도메인 API 동시 재설계
- 정산 로직까지 한 번에 MSA화

## 주요 리스크

- 조회 조합이 늘어나므로 읽기 모델 전략이 필요하다.
- 계정 생성과 도메인 생성 순서를 이벤트나 오케스트레이션으로 정의해야 한다.
- 전화번호 변경 같은 식별자 변경 이벤트의 소유권을 명확히 해야 한다.
- 운영 권한을 Account에 남길지 Ops 도메인으로 완전히 넘길지 결정이 필요하다.

## 우선 결정이 필요한 질문

1. 계정 하나가 여러 도메인 엔티티에 연결될 수 있는가
2. 배송원 계정 생성의 시작점은 Driver Profile 생성인가, Account 생성인가
3. 카카오 인증은 외부 인증 식별자 저장만 필요한가, 별도 인증 연동 서비스가 필요한가
4. 운영 권한은 Ops Staff 도메인 내부 RBAC로 둘지 별도 Authorization 서비스로 둘지
5. 차량 계정의 주 식별자는 차대번호인지 단말 식별자인지
