# Region Web First Slice Design

## 목적

이 문서는 권역 웹 1차를 `front-web-console` 기준으로 정의하고, 권역 정본과 권역 분석을 어떤 UX 흐름으로 묶을지 current truth로 고정한다.

이번 결정의 목적은 아래와 같다.

1. 권역 웹 1차를 `정본 관리 + 분석 조회` 한 묶음으로 고정한다.
2. `service-region-registry`와 `service-region-analytics`의 역할 경계를 유지한 채 단일 웹 흐름으로 연결한다.
3. 디자인 polish가 아니라 `운영 흐름 검증 가능한 UX`를 1차 목표로 둔다.
4. 권역 메뉴, 목록, 상세, 분석 확인 흐름을 구현 가능한 수준으로 내린다.

## 스코프

이번 설계에 포함하는 범위는 아래와 같다.

1. 권역 웹 1차의 정보 구조
2. 권역 목록과 권역 상세 흐름
3. 권역 정본의 생성/수정 범위
4. 권역 분석의 조회 범위
5. 최신 통계 요약의 노출 방식
6. 권한별 UI 노출 원칙

## 비스코프

이번 설계에 일부러 포함하지 않는 것은 아래와 같다.

1. 별도 권역 분석 대시보드
2. operator 전용 권역 웹
3. 분석 데이터 직접 생성/수정 UX
4. 권역 추천, route recommendation, GIS 고도화
5. dispatch, delivery, telemetry fan-in 자동화
6. 디자인 시스템 수준의 시각 polish

## 선택된 접근

이번 설계에서는 `권역 목록 -> 권역 상세 1페이지` 접근을 선택한다.

핵심 원칙은 아래와 같다.

1. 진입은 `권역 목록` 하나로 둔다.
2. 목록에서는 정본 정보만이 아니라 `최신 통계 요약`도 같이 본다.
3. 상세는 별도 페이지로 분리하고, 아래 네 영역을 한 화면에서 본다.
   - 기본 정보
   - polygon
   - 일별 통계
   - 기간 성과 요약
4. `service-region-registry`는 생성/수정 대상이다.
5. `service-region-analytics`는 1차에서 `읽기 중심`으로만 다룬다.

## 정보 구조

### 1. 메뉴

단일 웹 콘솔에 `권역` 메뉴를 추가한다.

### 2. 권역 목록

권역 목록은 아래 정보를 보여 준다.

1. 권역명
2. 상태
3. 난이도
4. 최신 일별 물량 요약
5. 최신 완료율 요약

원칙은 아래와 같다.

1. 목록은 `service-region-registry`의 정본 row를 기준으로 렌더링한다.
2. 요약 값은 `service-region-analytics`의 최신 snapshot을 붙여 보여 준다.
3. 목록 row는 상세 진입점 역할을 한다.
4. 목록 화면에서 긴 수정 폼을 열지 않는다.
5. 목록은 현재 존재하는 최신 요약 값만 보여 주고, `최신 기준일` 같은 추가 메타데이터 컬럼은 1차에서 노출하지 않는다.

### 3. 권역 상세

권역 상세는 아래 네 섹션으로 구성한다.

1. 기본 정보
2. polygon
3. 일별 통계
4. 기간 성과 요약

상세 진입 시 analytics는 바로 보여 준다.

원칙은 아래와 같다.

1. `일별 통계`, `기간 성과 요약`은 기본적으로 최근 데이터를 자동 표시한다.
2. 날짜/기간 필터는 선택 사항이다.
3. 사용자는 필터를 쓰지 않아도 권역의 최신 상황을 먼저 이해할 수 있어야 한다.

## 정본 관리 규칙

권역 정본은 `service-region-registry`가 계속 소유한다.

1차에서 가능한 작업은 아래와 같다.

1. 권역 생성
2. 권역 기본 정보 수정
3. status 변경
4. difficulty 변경
5. polygon 수정

polygon 입력은 1차에서 `raw GeoJSON textarea` 방식으로 유지한다.

이번 라운드에서 열지 않는 것은 아래와 같다.

1. hard delete
2. hierarchy 관리
3. polygon version history
4. bulk import/export

## 분석 조회 규칙

권역 분석은 `service-region-analytics`가 계속 소유한다.

1차 UX 원칙은 아래와 같다.

1. analytics는 `조회 중심`이다.
2. 상세에서 일별 통계와 기간 성과 요약을 바로 보여 준다.
3. 날짜/기간 필터는 선택 사항으로만 제공한다.
4. analytics를 직접 생성/수정하는 입력 흐름은 1차에서 열지 않는다.

## 권한 해석

권역 웹은 단일 웹 콘솔 안에서 권한별 UI 노출로 처리한다.

1차 권장 해석은 아래와 같다.

1. `system_admin`, `company_super_admin`는 권역 정본 수정 가능
2. `fleet_manager`, `settlement_manager`, `vehicle_manager`는 권역 화면을 읽기 중심으로 볼 수 있음
3. `driver` 웹 권한은 없다

즉 권역 웹은 `같은 라우트`를 쓰되, 권한에 따라 편집 액션만 달라진다.

## 서비스 경계

### `service-region-registry`

직접 소유하는 것:

1. `Region` 정본
2. 권역 코드, 이름, 상태, 난이도
3. polygon GeoJSON

소유하지 않는 것:

1. 권역 일별 통계
2. 기간 성과 요약
3. 분석 계산 로직

### `service-region-analytics`

직접 소유하는 것:

1. `RegionDailyStatistic`
2. `RegionPerformanceSummary`

소유하지 않는 것:

1. 권역 정본 수정
2. polygon
3. 권역 lifecycle

## front-web-console 1차 목표

권역 웹 1차는 아래 세 가지를 만족하면 닫힌다.

1. 관리자는 권역 목록에서 최신 운영 상태를 빠르게 파악할 수 있다.
2. 관리자는 권역 상세에서 정본 정보와 분석 결과를 한 문맥에서 볼 수 있다.
3. 정본 수정과 분석 확인이 별도 앱/별도 메뉴를 오가지 않아도 된다.

## 검증 기준

최소 검증 범위는 아래와 같다.

1. 권역 목록에서 registry row와 최신 analytics 요약이 함께 보인다.
2. 권역 상세에서 기본 정보, polygon, 일별 통계, 기간 성과 요약이 함께 보인다.
3. analytics는 필터 없이 기본 데이터를 즉시 보여 준다.
4. 날짜/기간 필터를 적용해도 같은 상세 흐름 안에서 결과가 갱신된다.
5. 권한이 없는 경우 수정 액션은 숨겨지고, 읽기 흐름은 유지된다.
