# Personnel Document Web First Slice Design

## 목적

이 문서는 단일 웹 콘솔에서 인사문서 1차를 어떻게 노출할지 current truth로 고정한다.

이번 설계의 목표는 아래와 같다.

1. `service-personnel-document-registry`를 단일 웹 콘솔의 독립 운영 메뉴로 연결한다.
2. 모든 인사문서가 반드시 `driver`와 연결된다는 원칙을 UI와 운영 흐름에 반영한다.
3. 디자인 polish가 아니라 `목록 -> 상세 -> 등록/수정`이 막히지 않는 운영 UX를 1차 목표로 둔다.
4. 파일 바이너리, 승인 워크플로우, 회사 aggregate를 끌어오지 않고 문서 메타데이터 관리만 웹에서 닫는다.

## 스코프

이번 설계에 포함하는 범위는 아래와 같다.

1. 단일 웹 콘솔의 `인사문서` 독립 메뉴
2. 인사문서 목록, 상세, 등록, 수정
3. `driver` 연결 규칙과 `driver_id` 필터
4. 권한별 읽기/수정 범위
5. `service-personnel-document-registry` 정본과 `service-driver-profile` read lookup 경계

## 비스코프

이번 설계에서 일부러 제외하는 것은 아래와 같다.

1. 파일 바이너리 업로드와 다운로드
2. 문서 미리보기
3. 승인/결재 워크플로우
4. 문서 만료 알림 자동화
5. 회사 단위 인사문서 aggregate
6. driver self-service 문서 제출

## 선택된 접근

이번 1차는 `독립 메뉴 -> 문서 목록 -> 문서 상세 -> 등록/수정` 접근을 선택한다.

핵심 원칙은 아래와 같다.

1. 시작점은 `기사 상세 하위 탭`이 아니라 단일 웹 콘솔의 독립 메뉴 `인사문서`다.
2. 모든 문서는 반드시 `driver_id`를 가진다.
3. 문서 목록은 문서 중심으로 보되, 기사 문맥으로 좁혀볼 수 있어야 한다.
4. 문서 정본은 계속 `service-personnel-document-registry`가 소유한다.
5. 기사 이름과 기본 문맥은 `service-driver-profile` read lookup으로 보강한다.

## 정보 구조

### 1. 메뉴

단일 웹 콘솔에 `인사문서` 메뉴를 추가한다.

### 2. 문서 목록

문서 목록은 아래 정보를 보여 준다.

1. 기사명
2. 기사 식별자
3. 문서 종류
4. 상태
5. 제목
6. 발급일
7. 만료일

원칙은 아래와 같다.

1. 목록 row의 정본 데이터는 `service-personnel-document-registry`를 기준으로 렌더링한다.
2. 기사명은 `driver_id`를 `service-driver-profile`과 조인하는 read lookup으로 보강한다.
3. 기본 정렬은 현재 서비스 정렬을 따르되, 1차는 목록 가시성을 우선한다.
4. `driver_id` 기준 필터를 제공해 기사 문맥으로 바로 좁혀볼 수 있어야 한다.

### 3. 문서 상세

문서 상세는 아래 섹션으로 구성한다.

1. 기사 연결 정보
2. 기본 문서 정보
3. 발급/만료 정보
4. 외부 참조와 메모
5. payload 원문

원칙은 아래와 같다.

1. 파일 뷰어 대신 메타데이터 중심으로 보여 준다.
2. payload는 1차에서 숨기지 않고 운영자가 raw 값으로 확인할 수 있게 둔다.
3. 상세 상단에서 수정으로 바로 이동할 수 있어야 한다.

### 4. 등록/수정

등록/수정 화면은 아래 필드를 다룬다.

1. `driver_id`
2. `document_type`
3. `status`
4. `title`
5. `document_number`
6. `issuer_name`
7. `issued_on`
8. `expires_on`
9. `external_reference`
10. `notes`
11. `payload`

원칙은 아래와 같다.

1. `driver_id`는 필수다.
2. 1차는 `driver_id`를 직접 선택하거나 전달받은 값으로 채우는 수준으로 유지한다.
3. 파일 업로드 필드는 두지 않는다.
4. 수정은 정본 메타데이터만 다룬다.

## 기사 연결 규칙

인사문서 1차는 `driver`와의 연결을 강하게 가진다.

1. 모든 문서는 반드시 하나의 `driver_id`에 속한다.
2. 문서 없는 기사도 있을 수 있지만, 기사 없는 문서는 없다.
3. 문서 화면은 독립 메뉴로 시작하되, 나중에 기사 상세에서 해당 기사 문서로 이동하는 링크를 추가할 수 있다.

## 권한 해석

1차 권한 원칙은 아래와 같다.

1. `system_admin` 생성/수정 가능
2. `company_super_admin` 생성/수정 가능
3. `settlement_manager` 생성/수정 가능
4. `fleet_manager` 생성/수정 가능
5. `vehicle_manager` 읽기만 가능
6. `driver` 웹 권한 없음

즉 `인사문서`는 정산/인력 운영 성격이 강한 도메인으로 해석한다.

## 서비스 경계

### `service-personnel-document-registry`

직접 소유하는 것:

1. 문서 메타데이터 정본
2. 문서 종류와 상태
3. 발급/만료/메모/외부 참조/payload

소유하지 않는 것:

1. 기사 기본 프로필
2. 바이너리 파일
3. 승인 워크플로우

### `service-driver-profile`

직접 소유하는 것:

1. 기사 기본 프로필 정본
2. 기사명과 회사/플릿 문맥

인사문서 웹에서의 역할:

1. `driver_id`를 기사명으로 보강하는 read source
2. 기사 문맥 필터의 기준 데이터

## 1차 목표

인사문서 1차는 아래가 되면 닫힌다.

1. 운영자는 인사문서 목록에서 기사별 문서를 파악할 수 있다.
2. 운영자는 문서 상세에서 메타데이터 전체를 확인할 수 있다.
3. 권한 있는 관리자는 문서를 생성/수정할 수 있다.
4. 기사와 문서의 연결이 끊기지 않는다.

## 검증 기준

최소 검증 범위는 아래와 같다.

1. 메뉴에 `인사문서`가 노출된다.
2. 문서 목록이 기사명/기사 식별자/문서 상태를 함께 보여 준다.
3. `driver_id` 필터가 동작한다.
4. 문서 상세에서 기사 연결과 문서 메타데이터를 함께 볼 수 있다.
5. 권한 없는 역할은 수정 액션이 숨겨지고 읽기만 가능하다.
