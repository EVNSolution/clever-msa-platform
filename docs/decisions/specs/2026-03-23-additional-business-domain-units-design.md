# Additional Business Domain Units 디자인

## 목적

이 문서는 현재 bootstrap platform 위에 추가로 계획해야 하는 `큰 업무 단위`를 고정하기 위한 설계 문서다.

이번 설계의 목표는 세 가지다.

1. 현재 이미 잡힌 코어 단위와 아직 비어 있는 큰 단위를 구분한다.
2. 아직 repo 이름이나 세부 서비스 이름이 정해지지 않은 영역을 큰 단위 수준에서 먼저 고정한다.
3. 향후 service skeleton 생성 전까지는 이 문서를 상위 정본으로 사용한다.

## 스코프

이번 설계에 포함하는 범위는 아래와 같다.

1. 현재 플랫폼에 이미 있는 큰 업무 단위
2. 추가 계획이 필요한 큰 업무 단위
3. 각 큰 단위의 현재 역할
4. 각 큰 단위의 미래 역할
5. 큰 단위와 기존 서비스 경계의 관계

## 비스코프

이번 설계에 포함하지 않는 것은 아래와 같다.

1. 각 큰 단위의 최종 repo 이름
2. 세부 서비스 개수와 세부 decomposition
3. 구체 API 설계
4. 테이블 스키마
5. 구현 순서 상세
6. service-local 문서 승격 시점

## 현재 이미 있는 큰 업무 단위

현재 platform에서 이미 bootstrap이 잡힌 큰 업무 단위는 아래와 같다.

- 계정
- 조직
- 배송원
- 차량
- 배정
- 단말
- 텔레메트리
- 정산

이 단위들은 이미 active runtime repo 또는 shell repo 수준으로 경계가 고정돼 있다.

## 추가 계획할 큰 업무 단위

이번에 추가로 계획 대상으로 고정하는 큰 업무 단위는 아래 다섯 개다.

- 배차
- 인사문서
- 권역분석
- 알림
- 공지 / 지원

핵심 원칙은 아래와 같다.

1. 이 다섯 개는 현재 서비스 개수 부족으로 보는 것이 맞다.
2. 세부 서비스 이름은 아직 고정하지 않는다.
3. 큰 단위별 역할만 먼저 정하고, skeleton 생성은 이후 단계에서 한다.
4. 상위 문서는 이 큰 단위의 정본이고, service-local 문서는 skeleton 생성 이후에만 검토한다.

## 1. 배차

### 현재 역할

`배차`는 미래 또는 당일 운영 계획을 다루는 큰 업무 단위다.

이 단위는 아래 질문에 답한다.

1. 어느 날짜에 어떤 물량이 예정되어 있는가
2. 어느 플릿과 회차에 누구를 배치할 것인가
3. 휴무, 예외, 가용성은 어떻게 반영되는가
4. 차량과 기사의 계획성 스케줄은 무엇인가

### 미래 역할

향후에는 아래 축을 포함할 수 있다.

1. 물량 계획
2. 회차별 배차 계획
3. 기사 가용성 / 예외
4. 차량 일정
5. 근무 일정

### 기존 서비스와의 관계

이 단위는 현재의 `service-vehicle-assignment`와 다르다.

- `vehicle-assignment`
  - 현재 시점의 배정 정본
- `배차`
  - 날짜, 회차, 플릿 기준의 운영 계획 정본

따라서 배차 기능을 `service-vehicle-assignment` 안으로 밀어 넣지 않는다.

## 2. 인사문서

### 현재 역할

`인사문서`는 배송원 기본 프로필 바깥의 문서성 정본을 다루는 큰 업무 단위다.

이 단위는 아래 질문에 답한다.

1. 계약과 증빙 문서는 무엇인가
2. 면허, 자격, 계좌, 사업자등록 정보는 무엇인가
3. 문서 상태와 유효성은 무엇인가
4. 소속, 근무지, 계약 정보가 어떤 문서와 연결되는가

### 미래 역할

향후에는 아래 축과 연결될 수 있다.

1. 승인 / 결재
2. 정산
3. 조직
4. 배차

### 기존 서비스와의 관계

이 단위는 현재의 `service-driver-profile`와 다르다.

- `driver-profile`
  - 이름, 연락처, 기본 소속 같은 얇은 프로필
- `인사문서`
  - 계약, 증빙, 계좌, 사업자, 문서 상태 같은 문서성 정본

따라서 문서 aggregate를 `service-driver-profile` 안으로 다시 키우지 않는다.

## 3. 권역분석

### 현재 역할

`권역분석`은 공간 단위의 운영 분석을 다루는 큰 업무 단위다.

이번 단계에서 고정하는 범위는 아래 세 가지다.

1. 권역 polygon
2. 권역 난이도
3. 권역별 배송 통계

### 미래 역할

향후에는 아래 축으로 확장될 수 있다.

1. 권역 기준 마스터
2. 권역 분석 결과
3. 권역별 배송 성과 비교
4. 이후 지도/추천 기능과의 연결

### 기존 서비스와의 관계

이 단위는 `vehicle-operations-view`나 `driver-operations-view`의 조회 필드로만 흡수하지 않는다.

권역 polygon, 난이도, 통계는 자체 정본과 분석 lifecycle을 가지므로 별도 큰 단위로 둔다.

## 4. 알림

### 현재 역할

`알림`은 시스템이 사용자에게 전달하는 알림 채널의 큰 업무 단위다.

이번 단계에서 고정하는 범위는 아래와 같다.

1. 푸시 토큰 관리
2. 푸시 발송
3. 발송 로그
4. 일반 알림함

### 미래 역할

향후에는 아래 이벤트와 연결될 수 있다.

1. 결재
2. 정산
3. 공지
4. 지원
5. 운영 이벤트

### 기존 서비스와의 관계

이 단위는 `공지`나 `지원`과 다르다.

- `알림`
  - 시스템 전달 채널
- `공지`
  - 운영이 게시하는 공지 정본
- `지원`
  - 사용자가 등록한 티켓과 처리 흐름 정본

따라서 알림은 별도 큰 단위로 유지한다.

## 5. 공지 / 지원

### 현재 역할

`공지 / 지원`은 운영 커뮤니케이션 영역의 큰 단위다.

이번 단계에서 이 큰 단위 안에는 아래 두 축이 함께 포함된다.

1. 공지
2. 지원

### 미래 역할

구현 단계에서는 아래처럼 분리 전제로 본다.

1. `공지`
   - 운영 공지 게시, 게시 상태, 게시 범위
2. `지원`
   - 문의, 티켓, 응답, 처리 상태 추적

### 기존 서비스와의 관계

이 큰 단위는 `알림`과 분리한다.

- `공지 / 지원`
  - 정본 업무 데이터
- `알림`
  - 그 결과를 전달하는 채널

즉 게시/티켓과 전달 채널을 같은 서비스로 합치지 않는다.

## 계획 수준 규칙

이번 문서에서 고정하는 계획 수준 규칙은 아래와 같다.

1. 현재는 큰 단위만 고정하고 세부 repo naming은 보류한다.
2. service skeleton이 생성되기 전까지는 `docs/`가 단일 정본이다.
3. 이 다섯 단위는 기존 서비스 안으로 억지로 흡수하지 않는다.
4. 각 단위의 세부 decomposition은 skeleton 생성 직전에 다시 자른다.

## 완료 기준

1. 현재 이미 있는 큰 업무 단위와 추가 계획 대상이 구분된다.
2. 추가 계획 대상 다섯 개의 이름과 역할이 고정된다.
3. `배차`, `인사문서`, `권역분석`, `알림`, `공지 / 지원`을 기존 bootstrap 서비스 안으로 억지로 넣지 않는다는 원칙이 고정된다.
4. 이후 service skeleton 생성 전까지 이 문서를 상위 계획 정본으로 사용한다.
