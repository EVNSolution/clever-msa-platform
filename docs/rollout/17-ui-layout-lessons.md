# 17. UI Layout Lessons

## 문서 목적

이 문서는 2026-04-11 세션에서 반복 확인된 UI 요구를 `lessons.md`식으로 짧고 강한 규칙으로 고정한다.

이 문서는 개별 화면 spec이 아니다. 운영형 웹 콘솔에서 다시 반복하면 안 되는 레이아웃, 반응형, 버튼 정렬, 중복 UI 실수를 막기 위한 current truth다.

## 작성 원칙

1. 설명보다 규칙을 먼저 쓴다.
2. 장식 취향이 아니라 운영 화면 기준으로 판단한다.
3. 새 화면을 만들 때는 이 문서를 먼저 보고, 예외가 필요하면 spec에서 따로 승인받는다.

## 기록 타이밍

1. 레슨은 작업 중간에 쓰지 않는다.
2. 레슨은 `완료 신호`를 보낼 수 있을 만큼 검증이 끝난 뒤에만 쓴다.
3. 배포가 있는 작업이면, 레슨 기록은 배포 이후에만 한다.
4. 즉 실험 중 관찰, 임시 추정, 아직 뒤집힐 수 있는 UI 판단은 이 문서에 올리지 않는다.
5. 이 문서는 작업 메모장이 아니라 배포 후 살아남은 규칙만 남기는 current truth다.

## Lessons

### 1. 헤더는 페이지별 즉흥 스타일이 아니라 템플릿이다

1. 같은 도메인군의 헤더는 `PageLayout` 템플릿으로 맞춘다.
2. 관리/정산처럼 같은 계열 화면은 헤더 리듬을 공유해야 한다.
3. 페이지 첫 패널을 헤더처럼 쓰지 않는다. 헤더와 본문 패널은 구조적으로 분리한다.
4. 본문 요소는 헤더 바로 아래에 붙을 수 있어야 한다.

### 2. 운영 화면은 표와 작업이 중심이다

1. 표보다 툴바와 장식이 먼저 보이면 실패다.
2. 흰 패널, 회색 계열 텍스트, 얇은 보더를 기본으로 쓴다.
3. 연녹색 같은 강조색은 이득이 명확한 선택 상태에만 쓴다.
4. 페이지에 강조색이 필요 없으면 과감히 빼고 무채색으로 간다.

### 3. 중복 UI는 제거한다

1. 페이지 전체 설명이 있으면 패널 안 동일 설명은 지운다.
2. 상단 요약, 검색 옆 개수, 하단 범위 텍스트처럼 같은 정보를 여러 번 반복하지 않는다.
3. 선택된 필터를 다시 칩으로 반복 노출하지 않는다.
4. 사용자가 한 번 이해한 정보는 다른 영역에서 다시 강조하지 않는다.

### 4. 반응형은 폭만이 아니라 높이까지 포함한다

1. 컨테이너는 viewport를 무조건 꽉 채우지 않는다. 적절한 하단 여백을 남긴다.
2. 다만 리스트/워크벤치 컨테이너 내부에서는 footer가 컨테이너 최하단에 붙어야 한다.
3. 행이 짧아도 footer는 위로 끌려 올라오면 안 된다.
4. 행이 길면 본문만 스크롤되고 footer는 고정된 마지막 행처럼 남아야 한다.
5. 좁은 폭에서는 필터와 액션이 줄바꿈되어도 정렬 규칙이 깨지면 안 된다.
6. 반응형 검수는 width뿐 아니라 height도 같이 본다.

### 5. footer는 컨테이너의 footer다

1. table footer처럼 보이게 만들지 말고, 컨테이너 전체의 마지막 행으로 붙인다.
2. footer controls는 우측 하단에 모은다.
3. 페이지 수가 1이면 페이지 번호는 숨긴다.
4. 페이지 크기 선택은 클릭 가능한 인라인 선택기처럼 보여야 한다.
5. `15 / 30 / 50 / 100 / all` 같은 선택지는 plain text처럼 가볍지만 control이라는 신호는 분명해야 한다.

### 6. 버튼과 액션 정렬은 기능만 맞으면 끝이 아니다

1. 버튼은 같은 행에서 기준선이 맞아야 한다.
2. 드롭다운, 검색창, 생성 버튼은 서로 다른 높이와 정렬로 흔들리면 안 된다.
3. 헤더 툴바의 액션은 독립 패널처럼 뜨지 말고, 필터와 같은 문맥 안에서 정렬돼야 한다.
4. 하단 footer controls도 텍스트, 버튼, select의 baseline이 맞아야 한다.

### 7. 검색/필터는 과장하지 않는다

1. 검색 영역을 위해 별도 장식 컨테이너를 만들지 않는다.
2. 필터와 검색은 필요한 만큼만 한 줄에 나열한다.
3. 검색창 폭은 화면을 먹어치우지 않게 제한한다.
4. 필터 정보와 결과 수는 필요한 위치 한 곳에만 둔다.

### 8. 페이지별 임시 보정 대신 재사용 가능한 규칙을 남긴다

1. 한 페이지에서 맞춘 헤더/툴바/리스트 규칙은 템플릿으로 올린다.
2. 전역 CSS를 함부로 건드리지 않는다. 꼭 필요한 경우에도 적용 범위를 클래스 하나로 제한한다.
3. 다른 페이지가 깨질 가능성이 있으면 즉시 공통 컴포넌트와 스코프를 다시 본다.

## 이번 세션에서 바로 적용한 영역

1. `배송원` 목록 화면
2. `관리자 역할`
3. `메뉴 정책`
4. `회사 메뉴 정책`
5. `정산 처리` 상위 헤더

## 검수 기준

새 UI를 끝냈다고 말하기 전에 아래를 확인한다.

1. 데스크톱 width/height에서 footer가 의도한 최하단에 있는가
2. 좁은 width에서 필터와 버튼 baseline이 깨지지 않는가
3. 같은 정보를 두 번 이상 반복 노출하지 않는가
4. 강조색이 정말 필요한가, 아니면 빼는 게 맞는가
5. 본문보다 헤더/장식이 더 먼저 보이지 않는가
6. 공통 템플릿으로 올릴 수 있는데 페이지 전용 CSS로 때우고 있지 않은가
